Problem: Four monolithic flat files hold all desktop frontend responsibilities, resulting in low locality; adding a single UI tab requires modifying all four files, while domain package types are completely duplicated instead of imported.

Solution: Decompose files into self-contained feature subfolders under `src/areas//` (e.g. `companion`, `gateway`, `skills`), using a standardized `AreaContext` interface and direct type reuse from the domain package.

Wins:

  • Locality: All code for one area resides in a single folder.
  • Leverage: A uniform initialization interface drives all areas dynamically.
  • Deletion test: Deleting a feature requires dropping only its directory.
  • Locality: Zero type duplication between front-end and packages.
ADR Note: Aligns with ADR-0010 (Tab switching via CSS display toggle is preserved; DOM nodes remain mounted).

Problem: 18 workspace packages introduce significant coordination overhead. 6 of these are shallow modules (<150 lines) and `runtime` is a vestigial 28-line pass-through re-export shim.

Solution: Merge `runtime`, `config`, `health`, `pad-engine`, and `security` into a single deep `@agentsoul/core` package. Consolidate `memory`, `entity`, and `semantic` into `@agentsoul/memory`.

Wins:

  • Deletion test: Remove vestigial runtime package entirely.
  • Locality: Keep related mathematical/utility domains grouped close together.
  • Locality: Reduce workspaces count from 18 to 12.
  • Leverage: Unified tests and simpler configuration surface.

Problem: The `gateway` package is a monolithic file (2,488 lines) handling routing, provider adaptation, channel storage, cost metrics, circuit breaker logic, security audits, backups, and MCP servers. It also violates database boundaries by initializing its own SQLite tables inline.

Solution: Separate the god module into four distinct packages: `@agentsoul/gateway-server` (handles routing & adapters), `@agentsoul/gateway-channels` (channel CRUD & circuit breaker), `@agentsoul/gateway-cost` (estimated costs), and `@agentsoul/gateway-audit` (traffic metadata auditing). Consolidate all database tables under `@agentsoul/persistence` schema control.

Wins:

  • Locality: DB schema definitions are centralized back to persistence.
  • Locality: Domain logic (costing, failover policies) is isolated from the HTTP server routing shell.
  • Leverage: Reusable gateway channel libraries that can run independently in testing.

4. Unify the Duplicate Daily Memory Channels

                    classDef leak stroke:#dc2626,stroke-width:2px;

After: Consolidated Memory Sink

                  flowchart TD
                    MCP[mcp-adapter] -->|MemoryStore Interface| MEM[memory]
                    MEM -->|Writes to| T2[memory_entries Table]

Problem: Dual storage schemas exist for daily topic memory. `mcp-adapter` writes directly to `mcp_memory_records` using its own logic, completely bypassing `@agentsoul/memory`'s `memory_entries` system, creating invisible silos of duplicate data.

Solution: Eliminate `McpMemoryStore` and its SQLite table. Connect the MCP memory tools directly to the unified `MemoryStore` interface exposed by `@agentsoul/memory`.

Wins:

  • Locality: Memory reads and writes funnel through a single module.
  • Locality: Unified database schema removes redundant tables.
  • Leverage: Simple protocol translation inside the MCP adapter.

Top Recommendation

We recommend executing Candidate 1: Front-end Area Decomposition first. The monolithic 2000+ line UI files represent the largest cognitive friction for developers, and breaking them apart creates clean, independent feature directories immediately ready for further packaging and extension.
